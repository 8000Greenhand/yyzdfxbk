# 短视频素材工作台 V2 · 产品与数据规范

## 1. 目标

工作台不是资料陈列页，而是一个内容决策系统。它必须把用户收集到的市场素材、每日 AI 变化、可调用知识、成熟选题和发布结果连接起来。

最终链路：

```text
飞书链接 / 市场素材 / AI最新变化
        ↓
素材总览：保留完整原材料
        ↓
提取可用知识、爆款表达、待验证项
        ↓
AI编辑部：结合时效、搜索需求和市场供给做判断
        ↓
知识库：只保留以后能再次调用的资产
        ↓
选题库：组合成解决完整问题的内容包
        ↓
发布
        ↓
复盘库：用真实数据反向修正知识价值和选题判断
```

本阶段不做自动发布，不为了“全自动”增加不必要的 API 或 Codex 消耗。

---

## 2. 顶部模块与固定顺序

顶部只保留以下五个一级模块，顺序不得调整：

1. **素材总览**
2. **AI编辑部**
3. **知识库**
4. **选题库**
5. **复盘库**

搜索框为全局搜索，可搜索标题、主题、标签、工具名、Skill 名、问题、分析内容和来源。

不要新增“工作流库”“工具库”“搜索雷达”等第六个一级模块。它们应作为相应模块的筛选或子视图。

---

## 3. 素材总览

### 3.1 作用

保存用户从飞书或其他入口收集的完整市场素材，回答：

- 原内容讲了什么；
- 为什么可能有效；
- 是因为硬核知识、爆款表达，还是两者兼有；
- 有哪些资产值得进一步加工。

### 3.2 数据来源

读取：

- `items/**/meta.json`
- `items/**/transcript.md`
- `items/**/analysis.md`

优先读取 `analysis.md` JSON 中的 `schema_version`。V2 使用 `2.0`；旧数据按兼容模式显示。

### 3.3 页面结构

顶部统计：

- 素材总数；
- 新增未分析；
- 已分析；
- 产生正式知识；
- 产生待验证知识；
- 产生可制作选题；
- 仅保留素材。

筛选：

- 平台；
- 发布时间；
- 收藏目的：硬核知识 / 爆款样本 / 两者兼有 / 仅灵感；
- 资产路由：知识库 / 选题库 / 仅素材；
- 验证状态；
- 内容形式；
- 一级主题与标签。

列表卡片第一屏显示：

- 封面、标题、作者、平台、发布时间；
- 一句话价值；
- `source_intent`；
- 是否进入知识库、选题库；
- 最终推荐选题或“暂不制作”；
- 验证状态和待补任务数量。

点击详情后分为：

1. 原内容分析；
2. 可调用知识；
3. 爆款表达；
4. 最推荐内容包；
5. 原逐字稿与来源。

### 3.4 强制规则

素材总览可以保存所有内容，但不得把所有内容自动升级为正式知识或成熟选题。

## 4. AI编辑部

### 4.1 作用

AI编辑部是每日和每周的决策中心，不是新闻仓库。回答：

- 今天有没有普通用户必须知道的重大变化；
- 用户最近在主动搜索什么；
- 市场上哪些内容已经高度竞争；
- 今天最值得做什么，或者为什么今天不追热点；
- 哪些信息需要补拍、查证或实测。

### 4.2 子视图

#### A. 今日判断

- 今日是否有 A 级重大更新；
- 一个最优行动：追更新 / 做搜索答案 / 补实测 / 暂不发布；
- 为什么选它，不选其他内容。

#### B. 更新拆解

每条更新必须显示：

- 官方来源与时间；
- 具体改动位置；
- 新旧区别；
- 适用套餐、账号、地区；
- 模型、价格、额度和限制；
- 普通用户现在要改变什么；
- 事实 / 界面观察 / 实测 / 推断 / 待验证标记。

#### C. 搜索需求雷达

这是 AI编辑部的子视图，不是独立一级模块。

每张需求卡显示：

- 搜索词与词族；
- 月搜索指数、竞争指数、趋势等已有数据；
- 用户真实意图；
- 现有内容供给和高互动样本；
- 答案缺口；
- 与账号匹配度；
- 内容有效期；
- 建议进入选题库或仅观察。

核心规则：搜索词决定“讲什么”，推荐逻辑决定“怎么包装”。

#### D. 待验证任务

如：

- 某个额度是否共用；
- 某个 Skill 是否支持 Codex；
- 某个工具国内能否使用；
- 同一任务的多工具实测；
- 缺少的官方截图或录屏。

状态：待找来源 / 待安装 / 待实测 / 已确认 / 已放弃。

### 4.3 推荐存储路径

后续每日简报可写入：

`editorial/daily/YYYY-MM-DD.json`

V1 可先支持工作台读取已有文件；不要求自动任务无人值守写入 GitHub。

## 5. 知识库

### 5.1 定义

知识库只保存以后能再次调用的资产，不做 AI 百科。一个知识只有在以下至少一种情况下才有价值：

1. 可以直接拿来使用；
2. 可以直接帮助做选择；
3. 可以直接帮助创作；
4. 值得继续核验，但当前还不能正式使用。

### 5.2 前台四类视图

#### A. 直接使用

工具、Skill、插件、平台功能、模板、提示词、GitHub 项目、完整操作教程。

必须显示：

- 解决什么具体问题；
- 最适合谁；
- 怎么使用；
- 入口和依赖；
- 账号、代码、API、电脑开机要求；
- 费用、额度和限制；
- 是否已实测；
- 替代方案；
- 关联素材与工作案例。

#### B. 帮助决策

模型选择、工具对比、价格和额度、适用边界、是否值得付费、何时不该使用 AI 等。

必须先给结论，再给判断依据。

#### C. 帮助创作

保存具体爆款表达资产，而不是空洞方法论：

- 原标题与封面组合；
- 前 3 秒表达；
- 可复用结构；
- 为什么有效；
- 成立条件；
- 适用主题；
- 针对用户账号的具体迁移；
- 照搬风险；
- 来源数据。

#### D. 待验证

只有二手说法、缺入口、缺完整步骤或需要实测的资产。

### 5.3 后台标签

一张卡只保留一个规范实体，通过标签跨场景使用。标签包括但不限于：

- 模型与平台；
- 工具；
- Skill；
- 插件；
- 作图；
- 视频；
- PPT；
- 数据分析；
- 小红书；
- 商务BD；
- 知识管理；
- 自动化；
- 搜索需求。

不得因为一个“PPT Skill”同时属于 Skill 和 PPT 就复制两张卡。

### 5.4 去重与合并

优先使用 `knowledge_candidates[].dedupe_key`，默认由“规范化名称 + 资产类型”组成。

合并规则：

- 同一个实体的多个来源合并为一张卡；
- 保留所有来源和时间；
- 新来源只能补充或提高验证等级，不能无证据覆盖已确认事实；
- 冲突信息并列显示，并标记待核验；
- 不自动把 `pending` 升级为 `formal`；
- `material_only` 不进入知识库。

验证等级：待找来源 → 来源已确认 → 已实测 → 稳定可复用。

### 5.5 第一屏必须 3 秒看懂

每张知识卡第一屏只显示：

- 名称；
- 类型；
- 一句话用途；
- 解决的问题；
- 最适合谁；
- 当前验证等级；
- 下一步动作。

详情再展开步骤、门槛、限制、来源、替代方案和关联内容。

## 6. 选题库

### 6.1 定义

选题库不是标题清单，而是成熟内容项目。一个关键词不能自动成为一个选题；一个选题必须帮助用户完成一次完整决策或任务。

### 6.2 来源

- 素材总览中的 `topic_package`；
- AI编辑部的重大更新与搜索缺口；
- 知识库中可组合的资产；
- 复盘库证明有效的表达、人群和主题。

### 6.3 三种内容产品

- 完整说明书 `complete_guide`；
- 同题实测 `comparison_test`；
- 真实工作改造 `real_work_transformation`。

新闻不是单独类型，应加工成以上三种之一。

### 6.4 选题卡字段

第一屏：

- 最终标题；
- 内容产品类型；
- 目标用户；
- 用户任务；
- 核心结论；
- 当前状态；
- 为什么现在值得做；
- 缺失证据数量。

详情：

- 用户最担心什么；
- 看完后完成什么决定；
- 搜索与市场证据；
- 现有答案缺口；
- 封面主副标题；
- 前 3 秒；
- 完整章节骨架；
- 每段核心信息和证据；
- 截图、录屏和实测清单；
- 关联知识卡与爆款表达；
- 发布目标；
- 发布门禁。

状态：

`needs_evidence` 待补证据 → `ready` 可制作 → 制作中 → 已发布 → 已放弃。

没有证据或没有完整答案的内容不得进入“可制作”。

## 7. 复盘库

### 7.1 作用

复盘库不是播放量登记表，而是让整个系统越来越准。

### 7.2 记录字段

基础：

- 平台、发布时间、链接；
- 选题卡 ID；
- 标题、封面和开头；
- 内容产品类型；
- 目标人群；
- 使用的知识卡和表达资产。

数据：

- 曝光；
- 封面点击率（如有）；
- 3 秒留存（如有）；
- 平均观看时长与完播率；
- 点赞、收藏、评论、分享；
- 主页访问、关注；
- 搜索流量占比；
- 进入笔记的搜索词；
- 7/14/30 天持续曝光。

判断：

- 曝光高、点击低：标题或封面问题；
- 点击高、观看低：开头未兑现承诺；
- 观看好、收藏低：缺少可执行价值；
- 收藏高、关注低：内容有用但账号承诺不清；
- 主页访问和关注高：适合作为账号主线。

结论必须回写：

- 哪类搜索词值得继续；
- 哪种标题封面组合有效；
- 哪些知识更容易带来收藏；
- 哪类案例更能带来关注和信任；
- 下一次保留、修改或停止什么。

## 8. V2 数据读取规则

### 8.1 analysis.md

从 Markdown 中提取最后一个合法 JSON 代码块。

- `schema_version == "2.0"`：读取 `source_intent`、`asset_route`、`knowledge_candidates`、`expression_assets`、`topic_package`；
- 无 `schema_version`：按旧字段兼容展示，不自动生成正式知识；
- JSON 无法解析：显示错误标记，不让单条错误阻断全量同步。

### 8.2 旧数据迁移

旧 `tools_resources` 是长文本，不能可靠自动升级成正式知识。

处理策略：

- 旧分析继续在素材总览显示；
- 可将旧 `rewrite_best_topic` 作为“旧选题建议”展示；
- 不把旧长文本直接写入知识库；
- 只有重新按 V2 分析后，才生成正式知识候选和表达资产。

### 8.3 本地显示与持久化

V1：

- 读取 GitHub 内容；
- 在浏览器内聚合、去重和筛选；
- 用户点击同步后刷新数据；
- 不要求浏览器直接写回 GitHub。

如果已有本地状态（如制作中、已发布、人工验证），保存在独立本地数据文件或 localStorage，不能覆盖 GitHub 原始分析。

## 9. 页面体验要求

- 首屏优先展示判断和动作，不堆长文本；
- 每张卡第一屏 3 秒能看懂；
- 长分析采用折叠分区；
- 验证状态、准入状态和制作状态使用稳定标签；
- 全局搜索支持别名和标签；
- 支持按“最近新增、最近更新、待核验、可制作、已发布”排序；
- 显示来源和更新时间，避免过时知识被当作当前事实；
- 不删除原有素材同步和详情查看能力；
- 不破坏中文路径和现有 GitHub 同步逻辑。

---

## 10. 本次 Codex 改造验收标准

完成后必须满足：

1. 顶部五个模块名称和顺序正确；
2. 现有素材仍能正常同步和查看；
3. V2 素材详情能展示知识候选、表达资产和完整内容包；
4. 旧素材仍兼容，不因缺字段报错；
5. 知识库只纳入 `admission=formal|pending`，不纳入 `material_only`；
6. 知识卡可按用途、类型、场景和验证状态筛选；
7. 同一知识可合并多个来源，不重复堆卡；
8. AI编辑部包含今日判断、更新拆解、搜索需求雷达和待验证任务；
9. 选题库只把 `topic_package.status=ready|needs_evidence` 纳入，显示完整骨架和缺失证据；
10. 复盘库具备录入、筛选和回看能力；
11. 同步失败、单条 JSON 错误和缺字段均有清晰提示；
12. 不接付费 API，不新增自动发布，不持续消耗 Codex 做日常内容分析。
