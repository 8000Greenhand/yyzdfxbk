# Codex 交接：短视频素材工作台 V2 改造

## 0. 先读哪些文件

改代码前必须先完整阅读：

1. `WORKBENCH_SPEC.md`：产品结构、数据规则和验收标准；
2. `ANALYSIS_PROMPT.md`：`analysis.md` V2 JSON 数据结构；
3. 当前仓库 `items/**/meta.json`、`transcript.md`、`analysis.md` 的真实样例；
4. 本地现有工作台代码，尤其是 GitHub 同步、Markdown JSON 解析、筛选、详情页和本地状态逻辑。

本次不是重新做一个新网页，而是在现有“短视频素材工作台”上升级。先审计，再改造。

---

## 1. 真实目标

把现有资料展示页升级为内容决策系统，固定为五个一级模块：

1. 素材总览
2. AI编辑部
3. 知识库
4. 选题库
5. 复盘库

信息流：

```text
市场素材/飞书链接/AI最新变化
→ 素材总览
→ 提取可调用知识、爆款表达、待验证项
→ AI编辑部结合时效、搜索需求和市场供给判断
→ 知识库沉淀可再次调用的资产
→ 选题库形成完整内容包
→ 发布
→ 复盘库反向修正判断
```

不要把页面做成五个互不相关的资料夹。

---

## 2. 不可破坏的现有能力

- 现有 GitHub 同步按钮必须继续可用；
- 现有素材、中文目录、封面、标题、作者、逐字稿和分析详情必须继续显示；
- 不修改 `transcript.md`；
- 不删除旧 `analysis.md`；
- 不接付费 API；
- 不做自动发布；
- 不把日常新闻抓取放进 Codex 定时任务；
- 不为了重构而重写整个项目；
- 新功能不能导致单条数据解析失败时全页崩溃。

先建立备份或可回滚提交，再开始改造。

---

## 3. 实施顺序

### 第一阶段：审计与数据层

先输出简短审计结论，但不要等待用户再次确认，确认没有破坏性风险后继续实施。

检查：

- 本地项目文件结构；
- 当前 GitHub 仓库地址和分支；
- 同步按钮实际读取哪些文件；
- `analysis.md` 的 JSON 提取方式；
- 是否已经使用 localStorage；
- 当前筛选、搜索、路由和详情页实现；
- 当前生成 `workbench.html` 的脚本或模板。

建立统一解析函数：

```text
parseMaterial(meta, transcript, analysis)
```

规则：

- 从 `analysis.md` 提取最后一个合法 JSON 代码块；
- `schema_version == "2.0"` 时读取 V2 字段；
- 没有 `schema_version` 时按 legacy 模式展示；
- JSON 解析失败只给该条素材错误标记；
- 缺字段一律安全降级，不允许页面报错；
- 原 Markdown 仍可在详情页阅读。

### 第二阶段：顶部导航

固定顺序：

`素材总览｜AI编辑部｜知识库｜选题库｜复盘库`

删除原“知识扩展”名称，将其能力并入“知识库”。

全局搜索框保留并升级，可搜索：标题、作者、标签、工具名、Skill 名、用户问题、选题、来源和分析内容。

### 第三阶段：素材总览

按 `WORKBENCH_SPEC.md` 实现。

必须支持：

- legacy 和 V2 同时显示；
- `source_intent`、`asset_route`、验证状态、选题状态筛选；
- V2 详情按“原内容分析 / 可调用知识 / 爆款表达 / 完整内容包 / 逐字稿与来源”分区；
- legacy 详情维持原有内容，并标记“旧分析，未生成结构化知识资产”。

素材重复处理：

- 优先按 `source_url` 分组；
- 没有 URL 时按“平台 + 规范化标题 + 作者”判断疑似重复；
- 不删除原文件；
- 列表默认显示一个主卡，并标明重复条数；
- 详情可展开查看各来源目录；
- 重复素材不得导致知识卡和选题卡重复生成。

### 第四阶段：知识库

知识库不是把 `tools_resources` 长文本直接展示。

数据只来自 V2：

```text
knowledge_candidates[]
```

纳入规则：

- `admission=formal`：进入正式视图；
- `admission=pending`：进入待验证；
- `admission=material_only`：不进入知识库；
- legacy `tools_resources` 只在素材详情展示，不自动升级。

去重：

- 以 `dedupe_key` 为主；
- 相同知识合并多个来源；
- 保留来源、时间、验证状态和冲突信息；
- 验证等级只能有证据地提升；
- 不自动把 `pending` 升为 `formal`。

前台四个用途视图：

1. 直接使用
2. 帮助决策
3. 帮助创作
4. 待验证

每张卡第一屏只显示：名称、类型、一句话用途、解决问题、适合谁、验证等级、下一步动作。其余信息折叠到详情。

### 第五阶段：选题库

数据来源：

- V2 `topic_package`；
- 后续 `editorial/daily/*.json`；
- 人工本地新增选题。

只纳入：

- `status=ready`；
- `status=needs_evidence`。

`do_not_make` 只在素材详情中显示，不进入正常选题列表。

选题不是标题卡，必须展示：

- 用户任务；
- 用户担心；
- 核心结论；
- 市场证据；
- 答案缺口；
- 标题、封面、前3秒；
- 完整章节骨架；
- 每段证据；
- 画面清单；
- 缺失信息；
- 发布目标和状态。

状态允许人工维护：

`needs_evidence → ready → producing → published → abandoned`

人工状态保存在 localStorage 或独立本地状态文件，不覆盖 GitHub 原始分析。

### 第六阶段：AI编辑部

先实现读取和展示，不实现自动抓新闻。

支持读取：

```text
editorial/daily/YYYY-MM-DD.json
```

没有文件时显示清晰空状态：

> 尚无编辑部简报。每日 AI 内容由 ChatGPT 定时任务生成，工作台只负责同步和展示。

子视图：

1. 今日判断
2. 更新拆解
3. 搜索需求雷达
4. 待验证任务

不得新增第六个一级模块“搜索雷达”。

### 第七阶段：复盘库

V1 先使用 localStorage 实现人工录入、编辑、导出和导入 JSON。

字段按 `WORKBENCH_SPEC.md`。

必须能够从“已发布选题”一键新建复盘，并记录：

- 发布链接和时间；
- 曝光、点击、观看、互动、主页访问、关注；
- 搜索流量与搜索词；
- 结论和下一步。

需要提供：

- 导出全部复盘 JSON；
- 导入复盘 JSON；
- 清晰提示本地数据存储位置和清空风险。

---

## 4. UI 原则

- 保留现有简洁白底蓝色体系，不做大面积重设计；
- 首屏优先显示判断、状态和下一步动作；
- 每张卡 3 秒看懂，不把长脚本直接平铺；
- 长内容折叠；
- 标签颜色稳定一致；
- 旧数据与 V2 数据有明显但克制的状态标记；
- 页面在 1366px 和 1920px 宽度下都可正常使用；
- 不引入不必要的大型框架或在线依赖；
- 本地直接打开 `workbench.html` 时仍能运行。

---

## 5. 测试要求

至少覆盖以下测试：

1. 一个 V2 `analysis.md` 正常显示所有结构；
2. 一个 legacy `analysis.md` 正常兼容；
3. 一个损坏 JSON 不影响其他素材；
4. 两个相同 `source_url` 的素材不会重复生成知识和选题；
5. 两个不同素材中的相同 `dedupe_key` 合并为一张知识卡；
6. `material_only` 不进入知识库；
7. `do_not_make` 不进入正常选题库；
8. 没有编辑部文件时显示空状态；
9. 复盘数据刷新页面后仍保留；
10. GitHub 同步后五个模块统计同步更新；
11. 中文路径和长标题不报错；
12. 本地直接打开页面不依赖开发服务器。
