# GPT 手动分析提示词

## 0. 固定触发方式

当用户在本项目里说类似以下指令时：

- “帮我处理这一批短视频”
- “处理新增短视频”
- “分析新增内容”
- “把这批内容写回仓库”

请自动执行本文件的完整流程，不要再要求用户重复解释分析标准。

完成后只需要在聊天里简短回复：已处理完成，写回了哪些条目；不要把完整分析内容贴回聊天，因为用户会通过飞书看板阅读结果。

---

## 1. 任务边界

请读取仓库里的 `items/**/meta.json` 和 `items/**/transcript.md`，只分析还没有有效 `analysis.md`/`analysis.json` 的条目。

不要修改 `transcript.md`。
不要上传视频或音频。
不要调用付费 API。
不要自动发布内容。
不要把视频/音频文件写进仓库。

目标不是普通总结，而是把外部视频/图文逐字稿转化为用户的小红书账号运营资产、学习复刻资产、工作流优化资产、Skill 素材和爆款结构库。

---

## 2. 分析总原则

用户发进来的内容通常默认是“用户认为有用”或“数据表现好”的内容，不要机械判断“有没有价值”。

分析重点是回答：

- 这条内容到底讲了什么？
- 它属于什么内容类型？
- 里面哪些部分可以被用户拿走？
- 它能不能被用户学习、复刻、改造成工作流、改造成小红书内容、沉淀成 Skill，或用于爆款结构参考？
- 如果适合改造成小红书内容，应该给出完整内容方案，而不是只给标题。
- 如果适合沉淀成 Skill 或工作流，必须写清楚名称、输入、输出、适用场景和使用方法。
- 如果具体工具/Skill 名称不是原视频明确提到的，不要编造；需要标注“需要额外检索确认”。

不要写空泛表述，例如：

- “对你有价值”
- “可以提升效率”
- “适合做内容”
- “可以做成 Skill”

必须具体说明：哪一段、哪一招、哪一个结构、哪一个流程、哪一个 Skill 机会可以被用户拿走。

---

## 3. analysis.md 输出结构

每个条目的 `analysis.md` 必须包含 Markdown 可读分析和一个 JSON 代码块。

Markdown 用于用户阅读。
JSON 用于飞书看板回写。

请按以下结构输出：

```md
# 分析结果

## 1. 内容分段整理

1. 开头痛点：
...

2. 背景/场景：
...

3. 核心方法：
...

4. 案例/流程展示：
...

5. 总结/引导：
...

> 如果逐字稿很长，请分成 5-8 段；如果较短，分成 3-5 段。目标是让用户不用读完整逐字稿，也能快速知道视频每一部分在讲什么。

## 2. 一句话总结

...

## 3. 内容基础分类

- 一级主题：AI工作流 / AI工具 / 小红书运营 / 达人BD / 职场效率 / 个人成长 / 商业认知 / 爆款案例 / 其他
- 内容形式：教程型 / 案例复盘型 / 工具盘点型 / 观点型 / 避坑型 / 情绪共鸣型 / 流程展示型
- 使用场景：...
- 原视频目标人群：...

## 4. 可用部分拆解

- 可复刻的流程：...
- 可借鉴的开头/标题/封面：...
- 可沉淀成 Skill 的部分：...
- 可改造成用户小红书内容的部分：...
- 可用于优化用户实际工作的部分：...

## 5. 利用方式判断

可多选：学习复刻 / 工作流改造 / 小红书内容改写 / 爆款结构参考 / Skill素材沉淀 / 普通归档

## 6. 爆款结构拆解

如果原内容数据明显好，或属于低粉高赞内容，请重点分析：

- 标题为什么可能有效：...
- 封面/开头如何吸引人：...
- 前 3 秒或开头钩子：...
- 内容结构：...
- 痛点/爽点/痒点：...
- 收藏、评论或转发动机：...
- 能否迁移到用户的 AI 职场提效账号：...

如果数据不明显或不是爆款样本，也要简单说明可借鉴结构。

## 7. 给用户的内容改造方案

如果这条内容可以被改造成用户的小红书内容，必须给出至少 1 个完整方案。

### 方案 A：...

- 主题：...
- 标题备选：
  1. ...
  2. ...
  3. ...
- 封面文案：...
- 开头钩子：...
- 内容结构：...
- 需要准备的材料：截图 / 录屏 / 数据 / 案例 / 工具 / 表格 / 前后对比
- 改造逻辑：这个方案借用了原视频的什么优势，又如何改成用户自己的真实场景

如有必要，可给方案 B，但不要为了凑数强行给多个方案。

## 8. Skill / 工作流沉淀

如果内容能沉淀成 Skill 或工作流，必须写清楚：

- 名称：...
- 来源：来自原视频 / 来自用户场景改造 / 需要额外检索确认
- 解决的问题：...
- 输入：...
- 输出：...
- 适合场景：...
- 如何用于用户的小红书账号、工作台或实际工作：...

## 9. 行业/工具补充

只有在确实有必要时才补充，不要为了写而写。

适合补充的情况：

- 原视频提到某个具体工具、Skill、模型、平台或自动化方法，但信息不完整；
- 该内容明显可以延伸到用户的小红书账号运营、AI 工作流、达人 BD、飞书看板、内容生产系统；
- 最近有新的工具/Skill/方法值得用户了解，且能具体说明对用户有什么用。

补充必须具体：

- 工具/Skill 名称
- 能解决什么问题
- 适合谁
- 可用于用户哪个实际场景
- 是否需要联网检索确认

不要写泛泛的行业趋势，不要写没有落地场景的知识点。

## 4. asset_level 判断标准

不要再用简单的“高/中/低”。改用：

- 核心资产：能学习复刻、改造工作流、发成用户内容、沉淀 Skill、指导爆款结构，至少满足其中两项。
- 可用素材：有明确可拿走的点，但不一定马上能用。
- 结构参考：内容方向和用户账号不强相关，但标题、开头、情绪或结构值得学习。
- 普通归档：暂时只保留，不进入主要使用视图。

---

## 5. 用户账号方向假设

除非用户后续明确更新，否则默认用户的小红书账号方向是：

普通职场人如何用 AI 把重复工作做成可复用工作流。

关键词：

- AI 落地到真实工作
- 普通职场人提效
- 本地/飞书/表格/图文/小红书工作台
- Skill 和工作流沉淀
- 不是单纯 AI 工具搬运
- 不是追热点，而是解决真实工作问题

内容改造时优先转成：

- 我用 AI 解决了一个真实工作问题
- 我把一个重复工作做成了工作流
- 我复刻了一个别人的 AI 流程，并改成适合我的版本
- 我拆了一个爆款，提炼出普通人也能用的方法
- 我把某个流程做成 Skill / 看板 / 工作台

---

## 6. 长期学习与沉淀机制

如果仓库中有多个已分析条目，请在分析新增内容时主动参考已有分析结果，逐渐识别：

- 哪类选题更适合用户账号；
- 哪类标题/封面/开头更值得借鉴；
- 哪些工作流反复出现，值得做成用户自己的 Skill；
- 哪些内容只是结构参考，不要强行改造成账号选题；
- 哪些内容适合形成系列化小红书栏目。

如有需要，可以维护或建议维护项目级沉淀文件，例如：

- `PROJECT_INSIGHTS.md`：沉淀阶段性爆款规律、账号选题方向、可复用工作流。
- `SKILL_IDEAS.md`：沉淀可开发的 Skill 清单。
- `CONTENT_IDEAS.md`：沉淀可拍的小红书选题。

但不要在未确认需求时强行新增太多文件。

---

## 7. 完成反馈

处理完成后，聊天里只需要简短反馈：

- 已处理完成；
- 写回了几个条目；
- 如有跳过，说明跳过原因，例如“已有有效 analysis.md”。

不要把完整分析贴回聊天。用户会从飞书看板查看。
