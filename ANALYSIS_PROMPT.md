# GPT 手动分析提示词 · V4

## 0. 固定触发方式

当用户在本项目里说类似以下指令时：

- “帮我处理这一批短视频”
- “处理新增短视频”
- “分析新增内容”
- “把这批内容写回仓库”

请自动执行本文件的完整流程，不要再要求用户重复解释分析标准。

完成后只需要在聊天里简短回复处理结果，不要把完整分析内容贴回聊天。用户会通过飞书看板阅读结果。

---

## 1. 任务边界

读取仓库里的 `items/**/meta.json` 和 `items/**/transcript.md`，只分析还没有有效 `analysis.md`/`analysis.json` 的条目；如果用户明确要求“重跑/覆盖”，则覆盖已有分析。

不要修改 `transcript.md`。
不要上传视频或音频。
不要调用付费 API。
不要自动发布内容。
不要把视频/音频文件写进仓库。

目标不是普通总结，而是把外部视频/图文逐字稿转化为用户的小红书账号运营资产、学习复刻资产、工具/技术资源、Skill 素材和爆款结构参考。

---

## 2. 分析核心原则

用户发进来的内容通常默认是“用户认为有用”或“数据表现好”的内容，不要机械判断“有没有价值”。

分析重点回答三件事：

1. 原视频/图文内容为什么有效？
2. 里面有哪些具体工具、技术、资源、方法值得拿走？
3. 它能不能改造成用户自己的小红书内容？如果能，怎么改？

不要写空泛表达，例如：

- “对你有价值”
- “可以提升效率”
- “适合做内容”
- “可以做成 Skill”
- “视频链接 → 逐字稿 → 分析 → 飞书看板”

最后一类是用户自己的系统基础流程，除非这条视频本身就是新启发，否则不要每条重复写。

---

## 3. analysis.md 输出结构

每个条目的 `analysis.md` 必须包含 Markdown 可读分析和一个 JSON 代码块。

Markdown 给用户阅读。
JSON 给程序解析和飞书看板同步。

请严格按以下结构输出：

```md
# 分析结果

## 1. 原视频内容分析

### 视频价值一句话

用一句话回答：这条视频/图文给谁看、解决什么问题、有什么价值。不要复述标题。

示例：
帮自媒体人批量跟踪对标账号，把新视频自动转成逐字稿，再拆成可复用的爆款规律和选题素材。

### 内容总结

用一句话讲清楚这条视频/图文讲什么。

### 标题分析

必须包含：
- 原始标题
- 标题为什么可能有效
- 它用了什么点击元素，例如数字、结果承诺、痛点、反常识、身份背书、情绪价值、工具名、强场景

### 开头原话

尽量提取原视频前 3 秒 / 前 5 秒的原话。
如果逐字稿不完整，就用“原视频核心开头可概括为：……”说明。

### 开头分析

不要只说开头讲了什么。要分析：
- 它为什么能吸引人；
- 打中了哪类用户；
- 用了结果前置、痛点前置、冲突前置、情绪安慰、提问自检还是反常识；
- 为什么用户愿意继续看。

### 内容结构分析

按原视频顺序拆结构。用 4-7 个步骤说明它怎么展开，不要写成完整逐字稿。

### 爆点分析

用资深小红书/抖音运营视角分析，不要只写“点击点、收藏点、评论点”表面结论。

要分析：
- 它真正传播的是什么，不只是表层主题；
- 它踩中了什么用户心理，例如效率焦虑、低成本诱惑、可复制幻想、情绪安慰、身份代入、结果承诺；
- 为什么用户会点、会收藏、会评论、会转发；
- 这条结构对用户账号有什么可迁移价值。

### 逐字稿

逐字稿不在此处重复展示，请由前端读取同目录下的 `transcript.md`。

## 2. 工具 / 技术 / 资源

只写有新增价值的内容。

每一个工具、技术、资源、方法必须写成：

- **名称**
  - 作用：它是做什么的。
  - 对你有什么用：它能如何帮助用户的小红书账号、工作台、实际工作或学习。

优先提取：
- 原视频明确提到的具体工具、模型、平台、开源项目；
- 原视频里有启发意义的方法；
- 用户还没有沉淀过、值得新增的 Skill 或检查表；
- 可用于进一步搜索学习的关键词。

如果原视频没有具体工具，就写可迁移的方法，不要硬写“无”。

不要重复写用户已有基础链路，例如：
- 视频链接 → 逐字稿 → 分析 → 飞书看板
- GitHub → GPT → 飞书
- 分析结果进入看板
- 素材入库流程

## 3. 给我的内容改造方案

如果这条内容可以改造成用户的小红书内容，给 1-2 个完整方案。

每个方案包括：
- 可改造方向
- 标题备选：3-5 个
- 封面文案：1-3 个
- 开头口播：前 3 秒/前 5 秒
- 内容结构：这条内容怎么拍/怎么写

### 可补充资料 / 延伸学习

这里不是写“用户需要准备什么材料”。

只写 GPT 可以补充的内容，例如：
- 可补充哪些概念解释；
- 可补充哪些工具资料；
- 可补充哪些搜索关键词；
- 可补充哪些参考案例；
- 建议学习什么。

如果没法补充，就不强行写。

## 4. JSON 字段要求

### card_value

用于中间卡片展示。必须表达“给谁、解决什么问题、有什么价值”。不要复述标题。

### title_analysis

必须包含原始标题和分析。

### opening_quote

尽量提取开头原话。不要把开头分析塞到这里。

### opening_analysis

必须分析前 3 秒/前 5 秒为什么吸引人，不要只概括内容。

### viral_points

必须用专业运营视角，分析传播心理和可迁移价值。

### tools_resources

必须“名称 + 作用 + 对用户有什么用”。不要只堆工具名。

### content_adaptation

必须是完整方案，而不是一句“可改造成某某选题”。

### primary_topic / content_format / scenario / target_user_in_video / use_modes / asset_level / tags

这些字段主要用于飞书筛选和分组，不要作为右侧详情主要阅读内容重复展示。

---

## 5. asset_level 判断标准

- 核心资产：能学习复刻、改造工作流、发成用户内容、沉淀 Skill、指导爆款结构，至少满足其中两项。
- 可用素材：有明确可拿走的点，但不一定马上能用。
- 结构参考：内容方向和用户账号不强相关，但标题、开头、情绪或结构值得学习。
- 普通归档：暂时只保留，不进入主要使用视图。

---

## 6. 用户账号方向假设

除非用户后续明确更新，否则默认用户的小红书账号方向是：

普通职场人如何用 AI 把重复工作做成可复用工作流。

关键词：

- AI 落地到真实工作
- 普通职场人提效
- 本地/飞书/表格/图文/小红书工作台
- Skill 和工作流沉淀
- 不是单纯 AI 工具搬运
- 不是追热点，而是解决真实工作问题

内容改造时优先转成：

- 我用 AI 解决了一个真实工作问题
- 我把一个重复工作做成了工作流
- 我复刻了一个别人的 AI 流程，并改成适合我的版本
- 我拆了一个爆款，提炼出普通人也能用的方法
- 我把某个流程做成 Skill / 看板 / 工作台

## 7. 完成反馈

处理完成后，聊天里只需要简短反馈：

- 已处理完成；
- 写回了几个条目；
- 如有跳过，说明跳过原因。

不要把完整分析贴回聊天。用户会从飞书看板查看。
