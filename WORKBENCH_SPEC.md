# AI内容创作工作台 V4.1 · 最终产品规范

## 0. 产品定位

这不是素材收藏夹，也不是批量仿写工具。

它是一套把外部信息转化为用户自己的认知与原创内容的工作台：

```text
看到外部内容或新变化
→ 判断它反映了什么真实需求
→ 提取并核查知识、包装和证据
→ 判断自己需要学什么、能不能讲
→ 组合成新的原创内容
→ 发布后用数据修正判断
```

最终目标是帮助用户持续创作更好的小红书内容，并逐步找到可以稳定涨粉的内容模型。

## 1. 不可偏离的原则

1. 原视频只是学习输入，不是最终交付；
2. 不从单条视频直接仿写一篇新稿；
3. 不把现有四条工作台视频当作账号永久边界；
4. 不强迫一个项目拆成 4—8 条重复内容；
5. 选题必须同时满足“用户需要、用户本人能学懂、2—4 小时内有机会完成”；
6. 普通素材默认轻分析，高价值素材才深分析；
7. 一份信息只存一个正式对象，不在多个页面重复复制；
8. 前台必须简洁，复杂推理留在后台；
9. 不接付费 API，不自动发布，不让 Codex 持续后台消耗额度；
10. 工作台打开后必须能告诉用户“今天先做什么”。

## 2. 账号内容边界

工作台服务的账号承诺暂定为：

> 替普通人试 AI、学 AI，把真正能用到工作和内容里的方法讲清楚。

它不是固定赛道，而是内容筛选标准。

允许进入的内容包括：

- 新 AI 知识和产品变化；
- Skill、工具、网站和开源项目；
- 工作与内容生产中的真实用法；
- 用户亲自做过的实验、工作台和失败复盘；
- 从爆款中发现、再经研究和实测补完整的用户问题。

禁止因为当前观众或现有四条视频而把账号锁死在“小红书运营、达人商务或工作台分享”。

## 3. 页面总体结构

页面标题：**AI内容创作工作台**。

副标题：**把外部信息变成自己的知识和原创内容**。

### 3.1 顶部数据条

只显示：

- 总收录链接：X 条；
- 飞书收录：X 条｜今日新增 X 条；
- 千瓜收录：X 条｜今日新增 X 条。

要求：

- 横向、小尺寸；
- 不占首屏主体；
- 来源无法确认时计入总数，但不得擅自归到飞书或千瓜；
- 点击来源可筛选素材；
- 今日新增按本地日期与可靠时间字段计算。

### 3.2 今日行动区

数据条下面放一个紧凑区域，不新增一级导航。

最多显示 3 项：

```text
今天先做什么
1. 可直接制作：某选题｜预计 2 小时
2. 需要实测：某工具｜预计 40 分钟
3. 待处理：3 条今日新增高价值素材
```

每项必须有明确按钮：

- 开始制作；
- 开始学习；
- 开始实测；
- 处理素材；
- 暂不处理。

没有可执行事项时，不制造任务，显示“今天暂无必须处理内容”。

### 3.3 五个一级板块

固定顺序：

1. **素材总览**
2. **AI编辑部**
3. **知识库**
4. **选题与学习**
5. **复盘库**

全局搜索支持标题、作者、分类、需求、Skill、工具、知识、选题、来源和平台。

## 4. 统一数据对象

后台只保留六类正式对象：

1. `material`：原始素材；
2. `demand`：用户需求；
3. `knowledge`：知识实体；
4. `packaging_reference`：真实标题封面参考；
5. `content_project`：内容项目；
6. `review`：发布数据与复盘。

规则：

- 同一需求只存一份，通过来源 ID 关联多条素材；
- 同一知识只存一份，通过来源列表合并；
- 包装参考必须指向真实素材，不复制成无来源模板；
- 选题页调用需求和知识，不重新复制一套文本；
- 页面是不同视图，不是六套互相独立的数据库。

V1 可以继续用 localStorage 持久化人工状态，但必须支持完整 JSON 导入、导出。若现有 GitHub 同步已支持写入独立状态文件，应复用，不另造第二套冲突逻辑。

推荐导出结构：

```text
workspace/
  demands.json
  knowledge.json
  packaging_references.json
  content_projects.json
  reviews.json
```

原始素材继续保留在 `items/**`，不得迁移或删除。

# 5. 板块一：素材总览

## 5.1 作用

处理飞书、千瓜和手动加入的原始链接，判断：

- 这条内容在讲什么；
- 它对我的创作有什么用；
- 是否值得进一步处理；
- 下一步应该进入需求、知识、包装、证据、选题还是归档。

## 5.2 左侧分类

默认分类：

- Skill / 工具；
- AI领域知识；
- 运营与内容；
- 职场效率；
- 个人成长；
- 包装参考；
- 待分类。

允许新增、修改和批量分类。

分类说明“内容讲什么”；价值标签说明“它对我有什么用”。

## 5.3 价值标签

一条素材可以多选：

- 需求证据；
- 知识线索；
- 包装参考；
- 表达参考；
- 证据素材；
- 补充已有主题；
- 仅保留。

## 5.4 素材列表

每条只显示：

- 封面；
- 原标题；
- 一句话内容；
- 已有点赞、收藏、评论、分享；
- 收录来源；
- 内容平台；
- 发布时间；
- 价值标签；
- 当前下一步。

不在卡片首屏堆长分析、完整脚本或复杂评分。

## 5.5 素材详情

顶部显示原链接、来源、平台、时间、数据与人工修改入口。

下面按价值条件显示，不强制每条都出现所有区块。

### A. 原始材料

- 封面；
- 标题；
- 完整逐字稿；
- 原始数据；
- 原 Markdown。

逐字稿默认折叠，支持展开和复制。

### B. 用户需求

最多显示：

- 核心需求；
- 需求证据；
- 最大答案缺口；
- 置信度。

核心需求必须具体到“谁、场景、阻碍、结果”。

### C. 可提取知识

默认最多展示 3 条：

- 名称；
- 一句话用途；
- 素材中说了什么；
- 状态：仅提及 / 已找到来源 / 已实测；
- 下一步核查。

点击后再显示来源、使用方法和确实影响使用的重要条件。

### D. 包装与表达

只展示最值得学的 1—2 点：

- 标题和封面如何分工；
- 为什么会停下或点击；
- 哪种表达可以迁移；
- 哪些不能照搬；
- 内容是否兑现承诺。

这里不批量生成标题和脚本。

### E. 对我的创作增量

回答：

- 它让我新增了什么：需求、知识、证据、表达还是判断；
- 当前能否讲清楚；
- 需要先学习、研究还是实测；
- 是否适合 2—4 小时制作条件。

### F. 下一步

只保留一个主要动作：

- 合并需求；
- 进入知识核查；
- 保存包装；
- 保存证据；
- 进入选题；
- 深度研究；
- 人工判断；
- 归档。

## 5.6 轻分析与深分析

默认读取 `analysis.md` V4.1 的轻分析结果。

只有以下情况显示“深度分析”：

- 用户手动点击；
- 素材数据优秀且材料充分；
- 同一需求有多条证据；
- 准备进入正式内容制作。

深度分析才允许调用完整爆款拆解 Skill。普通素材不得自动生成数页分析。

# 6. 板块二：AI编辑部

## 6.1 作用

AI编辑部不是新闻网站，也不是每天强制生成一篇稿子。

它负责：

- 发现近期值得了解的 AI 信息；
- 筛掉与账号无关的噪声；
- 识别可沉淀知识；
- 发现可测试的用户需求；
- 明确用户需要先学什么或实测什么。

## 6.2 信息来源

主要输入：

- AI HOT Skill；
- 官方博客、官方文档、GitHub Releases；
- 用户手动加入的链接；
- 素材库中反复出现的主题；
- 已发布内容的评论、搜索词和复盘结果。

AI HOT 用于发现，不作为最终事实源。所有准备进入知识库或内容项目的重要事实，必须回到原始链接核查。

AI HOT 处于公开测试阶段，工作台不能把核心运行依赖绑定在其长期在线上。获取失败时显示清楚的空状态，不影响其他板块。

## 6.3 更新方式

V1 不由 Codex 创建持续后台抓取任务。

提供一个“更新AI编辑部”入口，优先采用以下任一方式：

1. 生成可复制给支持 Skill 的 Agent 的完整指令；
2. 运行用户主动触发的本地更新脚本；
3. 导入 Agent 输出的 `editorial/daily/YYYY-MM-DD.json`。

不得在用户不知情时持续轮询或消耗模型额度。

## 6.4 页面内容

分两栏。

### 今日新增知识

每条只回答：

- 发生了什么；
- 它解决什么任务；
- 对哪类用户有用；
- 会改变什么选择或操作；
- 应更新哪条知识。

### 今日需求机会

每条只回答：

- 用户正在解决什么问题；
- 为什么判断有需求；
- 现有答案缺什么；
- 用户目前是否有能力制作；
- 建议：学习、研究、实测、制作或忽略。

页面操作：

- 加入知识库；
- 合并需求；
- 进入选题；
- 深度研究；
- 忽略。

不设置公开的“待验证大栏目”。未核查事实留在后台状态中。

# 7. 板块三：知识库

## 7.1 目标

知识库是一份可查、可用、持续合并的个人知识清单，不是长篇百科。

## 7.2 分类

- Skill；
- 工具与网站；
- AI产品与模型；
- AI领域知识；
- 运营与内容；
- 职场效率；
- 个人成长；
- 待分类。

## 7.3 列表

使用紧凑列表或表格，每条只显示：

- 名称；
- 一句话说明能解决什么；
- 类型；
- 来源数量；
- 最近更新时间；
- 状态：仅提及 / 已找到来源 / 已实测。

## 7.4 详情

按实际需要显示：

- 它是什么；
- 解决什么任务；
- 适合谁；
- 如何找到或开始；
- 会影响实际使用的重要条件；
- 原始来源；
- 用户是否亲自使用；
- 关联需求、素材和内容项目。

费用、API、地区、版权、账号、电脑要求等不是固定必填字段，只有影响决策时才显示。

## 7.5 合并规则

- 优先按规范名称和类型去重；
- 相同实体合并来源；
- 冲突信息并列并标明来源；
- 不能把“素材里提过”自动升级为“已确认”；
- 允许人工合并、拆分和改名；
- 相关知识在同一详情页下聚合，不额外制造庞大专题系统。

# 8. 板块四：选题与学习

## 8.1 核心作用

这里不只是生成稿子，还要先判断：

> 这个需求值得做吗？我现在能讲吗？不能讲，需要先学什么？

## 8.2 需求机会池

每条必须是完整用户问题，例如：

> 已经听说过 Codex、但不懂代码的人，想知道什么任务值得交给它。

不能只写“Codex教程”“AI工具”“小红书运营”。

每张卡显示：

- 用户问题；
- 需求来源与证据；
- 最大答案缺口；
- 当前准备状态；
- 预计制作成本；
- 下一步。

状态：

- 需求发现；
- 待学习；
- 待研究；
- 待实测；
- 可制作；
- 制作中；
- 已发布；
- 暂不适合。

## 8.3 制作门禁

进入内容生成前必须检查：

- 用户是否已经理解核心知识；
- 是否有可靠来源；
- 是否有真实操作、截图、案例或结果；
- 是否需要亲自实测；
- 是否能用自己的话讲清楚；
- 是否能在日常 2—4 小时完成。

输出制作成本卡：

```text
当前状态：需实测
预计总时间：2.5 小时
需要准备：官方资料 1 份、操作录屏 1 段、结果截图 2 张
推荐形式：60 秒录屏口播
```

用户做不了的选题不得直接生成完整稿。

## 8.4 选择知识和证据

直接关联知识库和素材，不复制文本。

可选择：

- 已核查知识；
- 用户本人实测；
- 官方资料；
- 真实截图；
- 数据和前后对比；
- 相关需求证据。

## 8.5 包装参考

从真实素材中选择 1—3 条参考：

- 原封面；
- 原标题；
- 数据；
- 它服务的需求；
- 本次借鉴什么；
- 本次不能照搬什么。

包装参考是市场样本，不是无来源标题模板。

## 8.6 内容生成

只有状态为“可制作”时生成内容任务包。

在不接模型 API 的 V1 中：

1. 工作台组装完整创作上下文；
2. 提供“复制到 ChatGPT / Codex”的按钮；
3. 用户把返回结果粘贴保存为内容项目；
4. 不允许关键词替换伪装成 AI 生成。

内容任务包包括：

- 本次给谁看；
- 解决什么问题；
- 用户看完得到什么；
- 知识、证据和个人经验；
- 参考包装及可迁移点；
- 一个主标题；
- 一组封面主副标题；
- 最多两组备用；
- 完整逐字稿；
- 每段画面；
- 剪辑标注；
- 发布前检查。

### 标题规则

- 20 个汉字是上限，不是越短越好；
- 标题必须语义完整；
- 多几个字能增强完整性和吸引力时，应充分利用空间；
- 允许适度夸张、情绪、冲突、悬念和结果放大；
- 不得编造功能、结果或用户无法兑现的承诺；
- 标题与封面必须互补，不机械重复；
- 不生成没对象、没场景、没结果的半句话。

### 封面规则

- 先解决 3 秒内是否看懂和想点，不只追求好看；
- 优先保留真实截图、成品和结果证据；
- 适配 3:4 小红书封面；
- 文字在缩略图中必须可读；
- 不强制套用某个 Skill 的固定风格，账号自己的视觉一致性优先。

## 8.7 剪辑范围

V4.1 不开发完整自动剪辑系统。

当前只生成：

- 竖屏画面规划；
- 哪段放大哪里；
- 哪句使用大字幕；
- 哪些等待和停顿删除；
- 哪段加速；
- 第一秒展示什么结果。

半自动剪辑工作台作为后续独立项目，不与本次改版绑定。

# 9. 板块五：复盘库

## 9.1 数据导入

支持：

- 小红书笔记明细表；
- 近 7 日观看数据；
- 手动补充截图中特有指标；
- 将数据绑定到内容项目。

## 9.2 单篇数据

记录：

- 曝光；
- 观看或阅读；
- 封面点击率；
- 2 秒退出；
- 5 秒完播；
- 平均观看时长；
- 完播率；
- 点赞、收藏、评论、分享；
- 主页访问；
- 新增关注；
- 流量来源；
- 搜索词；
- 观众画像。

自动计算：

- 收藏率；
- 分享率；
- 互动率；
- 平均观看占比；
- 每千次观看涨粉；
- 推荐流与搜索流比例。

## 9.3 四层复盘

每篇只回答：

1. 需求是否成立；
2. 包装是否有效；
3. 内容是否兑现；
4. 是否建立了关注理由。

同时输出：

- 下次保留什么；
- 下次只改一个什么；
- 下一条应该验证什么。

## 9.4 防止过度结论

- 样本少于 10 条时，只输出观察和假设；
- 不能根据当前 4 条内容锁死账号定位；
- 隐藏、删除、投放或曝光过低的内容标记为“非标准测试”；
- 不因单条低播放直接否定需求；
- 复盘结果反向更新需求、包装参考和制作门禁。

# 10. Skill 路由

所有 Skill 的安装地址、触发条件、输入输出和禁用规则，统一读取 `SKILL_ROUTING.md`。

静态网页本身不能假装直接调用 Agent Skill。

V1 的正确方式是：

- 工作台根据当前阶段组装任务；
- 显示应该调用的 Skill；
- 生成完整可复制指令；
- Agent 执行 Skill 后，将结构化结果导回工作台；
- 已有本地辅助服务时才允许做真实一键调用。

不得在前端用字符串替换伪装成 Skill 输出。

# 11. 防止成为高级收藏夹

必须执行以下约束：

1. 新素材默认轻分析；
2. 每条只给一个主要下一步；
3. 没有创作增量的素材直接归档；
4. 归档素材不在默认列表占据首屏，但保留搜索；
5. 首页只展示可执行事项，不展示大量未读信息；
6. AI编辑部没有重要信息时允许为空；
7. 选题不能仅因“热门”进入制作，必须通过能力和成本门禁；
8. 知识不以数量为目标，同一实体必须合并；
9. 不显示无意义的完成率和庞大待办数字制造焦虑；
10. 用户连续不使用的复杂功能应优先删除或降级，而不是继续增加功能。

# 12. 数据兼容

工作台必须兼容：

- V4.1：`schema_version == "4.1"`；
- V3：继续读取并映射旧字段；
- V2：继续兼容；
- legacy：无版本号分析仍可打开。

兼容规则：

- V3 的 `knowledge_assets` 映射为 V4.1 知识线索；
- V3 的 `viral_replication` 只保留原始包装参考，不把旧改写自动当成正式内容；
- V3 的 `topic_seeds` 作为候选需求，默认标记“需重新判断”；
- 从 `analysis.md` 读取最后一个合法 JSON 代码块；
- 单条错误不能导致全页失败；
- 原 Markdown 继续可查看。

# 13. UI 与技术要求

- 白底、深蓝或深青主色；
- 信息层级清楚，少用大卡片；
- 长逐字稿和长分析折叠；
- 1366px 与 1920px 均可正常使用；
- 本地直接打开 `workbench.html` 仍能浏览基础功能；
- 不引入不必要的大型框架；
- 不依赖易失效的在线前端资源；
- 所有人工修改必须持久化；
- 导入前校验，导出可回滚；
- GitHub 同步、中文路径和原素材不能被破坏。

# 14. 验收标准

1. 页面名称、数据条、今日行动区和五个板块正确；
2. 素材列表可按来源、分类和价值标签筛选；
3. V4.1 素材详情按实际价值条件展示，不强制填满；
4. 普通素材默认轻分析，高价值素材可发起深分析；
5. 每条素材只有一个主要下一步；
6. 用户需求可以合并多个来源；
7. 知识状态能区分仅提及、来源确认和本人实测；
8. 同一知识不会重复建卡；
9. AI编辑部可导入 AI HOT / Agent 结果，失败不影响工作台；
10. AI HOT 信息进入知识或创作前必须保留原始来源；
11. 需求机会卡显示准备状态和制作成本；
12. 用户做不了的选题不能直接生成完整稿；
13. 可制作项目能组装完整 ChatGPT / Codex 创作指令；
14. 标题规则、封面规则和真实参考调用正确；
15. V4.1 不开发完整自动剪辑，只输出剪辑标注；
16. 复盘可以导入基础数据并输出四层判断；
17. 少于 10 条内容时不输出永久定位结论；
18. V4.1、V3、V2、legacy 都能打开；
19. 单条损坏 JSON 不影响全页；
20. GitHub 同步、中文路径、逐字稿和旧素材不被破坏；
21. 不接付费 API、不自动发布、不创建 Codex 持续定时任务；
22. 工作台首屏能明确告诉用户今天最值得做什么；
23. 默认视图不会被归档素材和长分析淹没；
24. 所有 Skill 路由符合 `SKILL_ROUTING.md`；
25. 用户日常使用不需要先理解整套方法论。
