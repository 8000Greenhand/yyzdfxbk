# Codex 交接：AI内容创作工作台 V4.1 改造

## 0. 任务目标

把现有工作台从“素材收集与爆款改写工具”改造成：

> 把外部信息转化为用户自己的认知、原创内容和增长复盘的日常工作台。

这不是新做一个网页，也不是推翻现有导入和 GitHub 同步。

必须在现有工作台基础上改造，保留旧数据和旧能力，并让用户打开后能立刻知道“今天先做什么”。

## 1. 开始前必须完整阅读

按顺序阅读：

1. `WORKBENCH_SPEC.md`：V4.1 产品规范与验收标准；
2. `ANALYSIS_PROMPT.md`：V4.1 素材分析与 JSON 结构；
3. `SKILL_ROUTING.md`：各阶段 Skill 的安装、触发和禁用规则；
4. 当前仓库真实 `items/**/meta.json`、`transcript.md`、`analysis.md`；
5. 现有工作台代码、GitHub 同步、Markdown/JSON 解析和 localStorage。

已知本地工作台可能位于：

```text
D:\codex专用\飞书自动化\短视频收集助手\workbench.html
```

先确认真实路径和生成脚本。不要因为仓库中没有完整前端代码就另起一个不相干的新网页。

## 2. Skill 准备

按 `SKILL_ROUTING.md` 核查最新版 Skill。

优先安装：

```text
帮我安装这个 skill：https://aihot.virxact.com/aihot-skill/
帮我安装这个 skill：https://github.com/Cosmoslmj/cosmos-xiaohongshuvideo-breakdown
帮我安装这个 skill：https://github.com/ren644/xhs-title-skill
帮我安装这个 skill：https://github.com/ponyodong2026/ponyo-cover-anchor-system/tree/main/cover-anchor-system
帮我安装这个 skill：https://github.com/KKKKhazix/khazix-skills/tree/main/neat-freak
```

重型可选：

```text
帮我安装这个 skill：https://github.com/KKKKhazix/khazix-skills/tree/main/hv-analysis
```

不要把 `khazix-writer` 设为默认创作 Skill。

若某个 Skill 无法安装，不得卡住开发。必须保留“生成完整任务包 → 复制给 Agent → 导入结果”的降级路径。

## 3. 不可破坏的能力

- 保留现有 GitHub 同步按钮和同步逻辑；
- 保留飞书、千瓜和手动导入；
- 保留中文路径、封面、标题、作者、原链接、逐字稿和旧分析；
- 不修改 `transcript.md`；
- 不删除旧 `analysis.md`；
- 不批量重跑全部旧素材；
- 不上传媒体到 GitHub；
- 不接付费 API；
- 不自动发布；
- 不创建 Codex 持续定时抓取任务；
- 本地直接打开 `workbench.html` 仍可浏览基础数据；
- 单条坏 JSON 不能拖垮全页；
- 所有人工修改必须持久化并可导出回滚。

## 4. 先做代码与数据审计

开始编码前输出一份简短审计：

- 前端入口和生成脚本；
- GitHub 同步读取和写入哪些文件；
- 飞书、千瓜来源如何识别；
- `meta.json` 已有字段；
- `analysis.md` JSON 提取方式；
- 现有分类、收藏、选题和复盘保存在何处；
- localStorage 的 key 和数据结构；
- 是否存在重复数据源；
- 哪些旧能力可复用，哪些必须迁移。

先创建可回滚提交或备份，再开始修改。

# 5. 第一阶段：统一数据层

## 5.1 六类对象

内部统一为：

```text
material
demand
knowledge
packagingReference
contentProject
review
```

同一信息只存一份，页面通过 ID 关联。

不要让素材页、编辑部、知识库和选题库各自保存一份重复需求文本。

## 5.2 Material 解析

建立统一解析函数，例如：

```text
parseMaterial(meta, transcript, analysis, localState)
```

至少输出：

```text
id
sourceUrl
ingestionSource
platform
collectedAt
publishedAt
cover
title
author
metrics
transcript
rawAnalysisMarkdown
schemaVersion
primaryCategory
categoryLabel
tags
summary
cardValue
analysisDepth
useModes
demandSignal
knowledgeLeads
contentInsight
creatorTransfer
topicOpportunity
nextAction
parseError
```

## 5.3 版本兼容

- `schema_version == "4.1"`：原样读取新结构；
- `3.0`：兼容映射；
- `2.0`：兼容映射；
- 无版本号：legacy 安全展示；
- 从 `analysis.md` 读取最后一个合法 JSON 代码块；
- 缺字段使用安全默认值；
- 原 Markdown 始终可以查看。

V3 映射规则：

- `knowledge_assets` → `knowledgeLeads`；
- `viral_replication` → 包装参考，只保留真实原素材和分析，不把旧改写当正式成品；
- `topic_seeds` → 候选需求，默认状态“需重新判断”；
- 旧标题和脚本不删除，但不在首屏默认展示。

## 5.4 来源与时间

来源优先级：

1. `meta.json.ingestion_source`；
2. 现有导入链路中的可靠来源；
3. 用户人工修正；
4. 无法判断时 `unknown`。

不得把 unknown 误算到飞书或千瓜。

今日新增优先使用：

1. `collected_at`；
2. `exported_at`；
3. 可靠文件时间；
4. 无法确认则不计今日新增。

## 5.5 正式对象去重

需求：按完整用户问题和人工确认合并，保留来源素材 ID。

知识：按规范名称 + 类型去重，保留来源与状态。

包装参考：只指向真实素材 ID，不复制成无来源模板。

内容项目：关联需求、知识、证据和包装参考 ID。

复盘：关联已发布内容项目 ID。

# 6. 第二阶段：顶部与导航

页面标题改为：

```text
AI内容创作工作台
把外部信息变成自己的知识和原创内容
```

顶部数据条：

```text
总收录 X 条｜飞书 X 条 今日 +X｜千瓜 X 条 今日 +X
```

保持小尺寸。

增加紧凑“今天先做什么”区域，最多 3 项，只显示真实可执行任务。

导航固定：

```text
素材总览｜AI编辑部｜知识库｜选题与学习｜复盘库
```

全局搜索覆盖标题、作者、需求、知识、Skill、工具、选题、来源和平台。

# 7. 第三阶段：素材总览

## 7.1 左侧筛选

- Skill / 工具；
- AI领域知识；
- 运营与内容；
- 职场效率；
- 个人成长；
- 包装参考；
- 待分类；
- 用户自定义分类。

另支持价值标签筛选：

- 需求证据；
- 知识线索；
- 包装参考；
- 表达参考；
- 证据素材；
- 补充已有主题；
- 仅保留。

## 7.2 列表

每条只显示：

- 封面；
- 原标题；
- 一句话内容；
- 已有互动；
- 收录来源；
- 平台；
- 时间；
- 价值标签；
- 当前下一步。

归档素材默认不占首屏，但可搜索和筛选。

## 7.3 详情

按 V4.1 数据条件显示：

1. 原始材料；
2. 用户需求；
3. 可提取知识；
4. 包装与表达；
5. 对我的创作增量；
6. 下一步。

没有内容的区块不显示空壳。

逐字稿折叠、展开、复制。

## 7.4 轻分析与深分析

新素材默认只展示轻分析。

“深度分析”按钮：

- 生成 Cosmos Skill 任务包；
- 自动带入素材 ID、逐字稿、封面、标题、数据和项目覆盖规则；
- 不直接在前端伪造分析；
- 返回结果可粘贴或导入并绑定素材 ID。

普通素材不得自动调用完整深拆。

## 7.5 下一步动作

每条只能有一个主要动作：

```text
合并需求
进入知识核查
保存包装
保存证据
进入选题
深度研究
人工判断
归档
```

# 8. 第四阶段：AI编辑部

## 8.1 页面

两栏：

- 今日新增知识；
- 今日需求机会。

不恢复复杂新闻网站，不显示大批未核实条目。

## 8.2 AI HOT 接入

增加“更新AI编辑部”入口。

V1 优先实现：

1. 生成 AI HOT Skill 任务包；
2. 支持导入 `editorial/daily/YYYY-MM-DD.json`；
3. 若现有项目有本地 Python 辅助服务，可增加用户主动点击后执行的更新脚本；
4. 不创建后台定时任务；
5. 不让静态 HTML 直接依赖外部 API 才能打开。

任务包必须要求：

- 默认拉精选而不是全量；
- 根据时间、类别或关键词收窄；
- 按账号承诺过滤；
- 打开原始来源核查重要事实；
- 输出 `discovered|source_checked|tested` 状态；
- 给出 `ignore|knowledge|demand|research|topic` 动作。

AI HOT 不可用时显示空状态，不报全局错误。

## 8.3 操作

每条支持：

- 加入知识库；
- 合并需求；
- 进入选题；
- 深度研究；
- 忽略。

不公开展示庞大的“待验证”栏目。

# 9. 第五阶段：知识库

使用紧凑列表。

分类：

- Skill；
- 工具与网站；
- AI产品与模型；
- AI领域知识；
- 运营与内容；
- 职场效率；
- 个人成长；
- 待分类。

列表字段：

- 名称；
- 一句话用途；
- 类型；
- 来源数；
- 更新时间；
- 状态：仅提及 / 已找到来源 / 已实测。

详情按需显示入口、用法和真正影响使用的条件。

必须支持：

- 相同知识自动合并来源；
- 冲突信息并列；
- 人工合并、拆分、改名和分类；
- 关联素材、需求和内容项目；
- 二手提及不能自动变成已确认。

# 10. 第六阶段：选题与学习

## 10.1 需求机会卡

每条必须是完整用户任务。

显示：

- 用户问题；
- 需求证据；
- 答案缺口；
- 当前状态；
- 预计制作成本；
- 下一步。

状态：

```text
需求发现
待学习
待研究
待实测
可制作
制作中
已发布
暂不适合
```

## 10.2 制作门禁

检查：

- 用户是否理解；
- 来源是否可靠；
- 是否有真实证据和画面；
- 是否需要亲自实测；
- 是否能用自己的话讲清楚；
- 是否能在 2—4 小时完成。

显示制作成本卡。

未通过门禁时，只生成学习、研究或实测任务，不生成完整稿。

## 10.3 深度研究

“深度研究”动作生成 HV Analysis 任务包，并明确标记重型任务。

不得对普通短内容自动调用。

## 10.4 包装参考

选择 1—3 条真实素材：

- 原封面；
- 原标题；
- 数据；
- 对应需求；
- 借鉴点；
- 不可照搬点。

## 10.5 正式创作任务包

只有“可制作”时启用。

组装：

- 目标用户；
- 用户任务；
- 内容承诺；
- 已核查知识；
- 个人经验和真实感受；
- 证据和画面；
- 参考包装；
- 标题规则；
- 封面规则；
- 逐字稿要求；
- 剪辑标注要求。

提供：

- 复制到 ChatGPT；
- 复制到 Codex；
- 导出 Markdown；
- 粘贴/导入返回结果；
- 保存为内容项目。

## 10.6 标题 Skill

内容结构和核心承诺确定后，才能生成 XHS Title 任务包。

必须写入项目覆盖规则：

- 20 字以内是上限，不是越短越好；
- 语义完整；
- 允许适度夸张、情绪和悬念；
- 不编造结果；
- 标题与封面互补；
- 最终只保留 1 主 + 最多 2 备选。

## 10.7 封面 Skill

标题确定后，生成 Cover Anchor 任务包。

必须带入：

- 内容类型；
- 目标用户；
- 标题；
- 可用真实截图；
- 用户账号现有视觉偏好；
- 3:4 尺寸；
- 不强制套固定配色；
- 不添加他人水印和虚假数据。

## 10.8 剪辑

本次不开发完整自动剪辑。

内容项目只保存：

- 竖屏构图；
- 时间轴；
- 放大区域；
- 大字幕；
- 删除停顿；
- 等待画面加速；
- 第一秒结果画面。

不要在 UI 中声称已实现 AI 自动剪辑。

# 11. 第七阶段：复盘库

支持导入：

- 笔记明细表；
- 近 7 日观看数据；
- 手动补充指标。

绑定内容项目并计算：

- 收藏率；
- 分享率；
- 互动率；
- 平均观看占比；
- 每千次观看涨粉；
- 推荐流与搜索流比例。

每篇输出四层复盘：

1. 需求是否成立；
2. 包装是否有效；
3. 内容是否兑现；
4. 是否形成关注理由。

再给：

- 下次保留什么；
- 下次只改一个什么；
- 下一条验证什么。

规则：

- 少于 10 条只给观察与假设；
- 隐藏、删除、投放或曝光太低标记非标准测试；
- 不根据当前四条锁死账号定位。

# 12. 防止高级收藏夹

代码中落实：

- 普通素材轻分析；
- 每条只有一个主要下一步；
- 无创作增量直接归档；
- 归档不占默认首屏；
- 今日行动最多 3 项；
- AI编辑部允许空；
- 不用无意义的待办数量制造焦虑；
- 不自动生成大量选题；
- 同一知识和需求必须合并；
- 不把所有旧素材一次性升级成 V4.1 长分析。

# 13. UI 要求

- 白底、深蓝或深青主色；
- 页面干净、信息密度适中；
- 卡片首屏只放关键内容；
- 长文本折叠；
- 1366px、1920px 正常；
- 不引入不必要的大框架和在线依赖；
- 本地 HTML 可运行；
- 所有空状态清楚；
- 操作后有明确反馈；
- 不让用户先学习方法论才能用网页。

至少验证：

1. 顶部三组统计正确；
2. unknown 来源不误算；
3. 今日行动最多 3 项且可执行；

6. 单条坏 JSON 不影响全页；
7. 分类、价值标签和来源可人工修改并持久化；
8. 归档素材默认隐藏但可搜索；
