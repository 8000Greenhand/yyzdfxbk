# GPT 手动分析提示词 · V3

## 0. 固定触发方式

当用户在本项目里说类似以下指令时：

- “帮我处理这一批短视频”
- “处理新增短视频”
- “分析新增内容”
- “把这批内容写回仓库”

请自动执行本文件的完整流程，不要再要求用户重复解释分析标准。

完成后只需要在聊天里简短回复处理结果，不要把完整分析内容贴回聊天。用户会通过飞书看板阅读结果。

---

## 1. 任务边界

读取仓库里的 `items/**/meta.json` 和 `items/**/transcript.md`，只分析还没有有效 `analysis.md`/`analysis.json` 的条目，或在用户明确要求“重跑/覆盖”时覆盖已有分析。

不要修改 `transcript.md`。
不要上传视频或音频。
不要调用付费 API。
不要自动发布内容。
不要把视频/音频文件写进仓库。

目标不是普通总结，而是把外部视频/图文逐字稿转化为用户的小红书账号运营资产、学习复刻资产、工具/技术资源、Skill 素材和爆款结构参考。

---

## 2. 核心原则

用户发进来的内容通常默认是“用户认为有用”或“数据表现好”的内容，不要机械判断“有没有价值”。

分析重点是回答三件事：

1. 原视频内容为什么有效？
2. 里面有哪些具体工具、技术、资源、方法值得拿走？
3. 它能不能改造成用户自己的小红书内容？如果能，怎么改？

不要重复展示看板里已经有的基础字段，例如标题、平台、作者、发布时间、数据、标签。那些由前端中间卡片展示。

不要写空泛表述，例如：

- “对你有价值”
- “可以提升效率”
- “适合做内容”
- “可以做成 Skill”
- “视频链接 → 逐字稿 → 分析 → 飞书看板”

最后一类是用户自己的系统基础流程，除非这个流程本身是本条视频的新启发，否则不要每条重复写。

---

## 3. analysis.md 输出结构

每个条目的 `analysis.md` 必须包含 Markdown 可读分析和一个 JSON 代码块。

Markdown 给用户阅读。
JSON 给程序解析和飞书看板同步。

请严格按以下结构输出：

```md
# 分析结果

## 1. 原视频内容分析

### 内容总结

用一句话讲清楚这条视频/图文讲什么。不要再单独写“一句话总结”。

### 标题分析

分析原视频标题为什么可能有效。重点看：数字、结果承诺、痛点、反常识、身份背书、情绪价值、工具名、强场景。

### 开头分析

分析前 3 秒/前 5 秒如何吸引人。重点看：先展示结果、先打痛点、先制造冲突、先给情绪出口，还是先抛问题。

### 内容结构分析

按原视频顺序拆结构。用 4-7 个步骤说明它怎么展开，不要写成完整逐字稿。

### 爆点分析

拆它为什么可能数据好：
- 点击点
- 收藏点
- 评论点
- 转发点
- 真正有价值的核心

### 逐字稿

逐字稿不在此处重复展示，请由前端读取同目录下的 `transcript.md`。

## 2. 工具 / 技术 / 资源

只写有新增价值的内容。

优先提取：
- 原视频明确提到的具体工具、模型、平台、开源项目；
- 原视频里有启发意义的方法；
- 用户还没有沉淀过、值得新增的 Skill 或检查表；
- 可用于进一步搜索学习的关键词。

如果原视频没有具体工具，就写可迁移的方法，不要硬写“无”。

不要重复写用户自己系统已有流程，例如：
- 视频链接 → 逐字稿 → 分析 → 飞书看板
- GitHub → GPT → 飞书
- 分析结果进入看板
- 素材入库流程

## 3. 给我的内容改造方案

如果这条内容可以改造成用户的小红书内容，给 1-2 个完整方案。

每个方案包括：
- 可改造方向
- 标题备选：3-5 个
- 封面文案：1-3 个
- 开头口播：前 3 秒/前 5 秒
- 内容结构：这条内容怎么拍/怎么写

### 可补充资料 / 延伸学习

这里不是写“用户需要准备什么材料”。

只写 GPT 可以补充的内容，例如：
- 可补充哪些概念解释；
- 可补充哪些工具资料；
- 可补充哪些搜索关键词；
- 可补充哪些参考案例；
- 建议学习什么。

如果没法补充，就不强行写。

## 4. 字段说明

### summary

只写内容总结，不要和正文里再重复“一句话总结”。

### title_analysis / opening_analysis / structure_analysis / viral_points

用于右侧 Tab 1“原视频内容分析”。

### tools_resources

用于右侧 Tab 2“工具 / 技术 / 资源”。

只展示有新增价值的工具、技术、平台、开源项目、方法、Skill 或搜索关键词。

### content_adaptation / supplement_materials

用于右侧 Tab 3“给我的内容改造方案”。

不要写成用户任务清单。能补充资料就补充；不能补充就少写。

### primary_topic / content_format / scenario / target_user_in_video / use_modes / asset_level / tags

这些字段主要用于飞书筛选和分组，不要作为右侧详情主要阅读内容重复展示。

---

## 5. asset_level 判断标准

- 核心资产：能学习复刻、改造工作流、发成用户内容、沉淀 Skill、指导爆款结构，至少满足其中两项。
- 可用素材：有明确可拿走的点，但不一定马上能用。
- 结构参考：内容方向和用户账号不强相关，但标题、开头、情绪或结构值得学习。
- 普通归档：暂时只保留，不进入主要使用视图。

---

## 6. 用户账号方向假设

除非用户后续明确更新，否则默认用户的小红书账号方向是：

普通职场人如何用 AI 把重复工作做成可复用工作流。

关键词：

- AI 落地到真实工作
- 普通职场人提效
- 本地/飞书/表格/图文/小红书工作台
- Skill 和工作流沉淀
- 不是单纯 AI 工具搬运
- 不是追热点，而是解决真实工作问题

内容改造时优先转成：

- 我用 AI 解决了一个真实工作问题
- 我把一个重复工作做成了工作流
- 我复刻了一个别人的 AI 流程，并改成适合我的版本
- 我拆了一个爆款，提炼出普通人也能用的方法
- 我把某个流程做成 Skill / 看板 / 工作台

---

## 7. 完成反馈

处理完成后，聊天里只需要简短反馈：

- 已处理完成；
- 写回了几个条目；
- 如有跳过，说明跳过原因，例如“已有有效 analysis.md”。

不要把完整分析贴回聊天。用户会从飞书看板查看。
