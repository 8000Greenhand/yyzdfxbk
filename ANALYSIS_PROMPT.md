# GPT 手动分析提示词 · 爆款脚本标准版

## 0. 固定触发方式

当用户在本项目里说类似以下指令时：

- “帮我处理这一批短视频”
- “处理新增短视频”
- “分析新增内容”
- “把这批内容写回仓库”
- “按最新内容标准重跑”

请自动执行本文件的完整流程，不要再要求用户重复解释分析标准。

完成后只在聊天里简短回复处理结果，不要把完整分析内容贴回聊天。用户会通过飞书看板阅读结果。

---

## 1. 任务边界

读取仓库里的 `items/**/meta.json` 和 `items/**/transcript.md`，分析还没有有效 `analysis.md` 的条目；如果用户明确要求“重跑/覆盖”，则覆盖已有分析。

不要修改 `transcript.md`。
不要上传视频或音频。
不要调用付费 API。
不要自动发布内容。

目标不是普通总结，也不是标题党式改写，而是把参考爆款完整拆解后，迁移成适合用户账号的高信任小红书脚本。

---

## 2. 用户账号默认定位

除非用户后续明确更新，否则默认用户的小红书账号方向是：

**普通职场人如何用 AI 把重复工作做成可复用工作流。**

表达原则：

- 不卖课
- 不神化 AI
- 不讲暴富
- 不做单纯工具搬运
- 讲真实工作场景
- 讲具体流程、Skill、坑点、边界和复盘
- 让用户学到“怎么把 AI 用进自己的重复工作”

---

## 3. 分析核心原则

用户发进来的内容通常默认是“用户认为有用”或“数据表现好”的内容，不要机械判断“有没有价值”。

分析重点回答四件事：

1. 原视频/图文为什么有效？
2. 它有哪些具体工具、技术、资源、方法值得拿走？
3. 参考爆款的底层逻辑是什么？
4. 如果这是用户自己的账号，应该怎么重写成一条更可能涨粉/收藏/建立信任的小红书脚本？

禁止只写：

- “可以改造成某某选题”
- “适合做内容”
- “可以提升效率”
- “标题吸引人、内容有价值”
- “AI工作台很有用”

必须写清楚：

- 针对谁
- 解决什么问题
- 提供什么价值
- 参考爆款靠什么成立
- 迁移到用户账号后怎么讲才可信
- 完整脚本怎么说

---

## 4. analysis.md 输出结构

每个条目的 `analysis.md` 必须包含 Markdown 可读分析和一个 JSON 代码块。

Markdown 给用户阅读。
JSON 给程序解析和飞书看板同步。

请严格按以下结构输出：

```md
# 分析结果

## 1. 原视频内容分析

### 视频价值一句话

用一句话回答：这条视频/图文给谁看、解决什么问题、有什么价值。不要复述标题。

### 内容总结

用一句话讲清楚这条视频/图文讲什么。

### 标题分析

必须包含：
- 原始标题
- 标题为什么可能有效
- 它用了什么点击元素，例如数字、结果承诺、痛点、反常识、身份背书、情绪价值、工具名、强场景

### 开头原话

尽量提取原视频前 3 秒 / 前 5 秒的原话。
如果逐字稿不完整，就用“原视频核心开头可概括为：……”说明。

### 开头分析

不要只说开头讲了什么。要分析：
- 它为什么能吸引人；
- 打中了哪类用户；
- 用了结果前置、痛点前置、冲突前置、情绪安慰、提问自检还是反常识；
- 为什么用户愿意继续看。

### 内容结构分析

按原视频顺序拆结构。用 4-7 个步骤说明它怎么展开。

### 爆点分析

用资深小红书/抖音运营视角分析：
- 它真正传播的是什么；
- 它踩中了什么用户心理；
- 为什么用户会点、会收藏、会评论、会转发；
- 它如何建立信任；
- 它对用户账号有什么可迁移价值。

## 2. 工具 / 技术 / 资源

只写有新增价值的内容。

每一个工具、技术、资源、方法必须写成：

- **名称**
  - 作用：它是做什么的。
  - 对你有什么用：它能如何帮助用户的小红书账号、工作台、实际工作或学习。

不要重复写用户已有基础链路，例如：
- 视频链接 → 逐字稿 → 分析 → 飞书看板
- GitHub → GPT → 飞书
- 分析结果进入看板
- 素材入库流程

## 3. 爆款脚本重写方案

### 参考爆款逻辑拆解

必须分析参考爆款为什么成立，不要只抄标题和开头。

要写清楚：
- 它给了什么强结果；
- 它打中了什么真实痛点；
- 它如何建立信任；
- 它讲了哪些工具、Skill、流程、坑点、限制或判断标准；
- 它为什么让人愿意收藏或追问；
- 它有哪些逻辑能迁移，哪些不能照搬。

### 我的账号迁移判断

必须判断：
- 放到用户账号里应该怎么变；
- 目标用户是谁；
- 核心痛点是什么；
- 用户看完能学到什么；
- 这条内容适合涨粉、收藏、建立信任还是展示工作台能力；
- 是否符合“普通职场人用 AI 解决真实工作问题”的人设。

### 我会押的爆款选题

只给一个最推荐选题。不要给一堆方向让用户自己选。

如果这个字段会作为小红书标题、卡片标题或选题标题展示，必须控制在 **20 字以内**，优先 8-16 字。不要写完整长句，不要把脚本主题句当标题。

### 标题备选

给 3-5 个标题，并保证标题承诺和正文内容匹配。

**硬性规则：每个标题必须不超过 20 字，优先 8-16 字。** 中文、英文、数字、标点都计入展示长度。标题要短、准、有钩子；不要把完整观点句塞进标题。长标题必须压缩成痛点、反差、结果或动作。

标题可以优先使用这些形式：
- 痛点型：收藏越多越写不出
- 反差型：收藏夹不是素材库
- 动作型：好视频要拆成选题
- 结果型：用AI先拆素材
- 避坑型：别再只收藏爆款了

### 封面文案

只写主标题和副标题，不写画面素材建议。

### 完整口播脚本

必须给可直接拍的完整口播稿。可以按时间段写，也可以写成自然口播，但必须包含：
- 前 3 秒钩子；
- 真实痛点；
- 具体场景；
- 参考爆款里可迁移的方法逻辑；
- 工具 / Skill / 流程细节；
- 遇到的问题或避坑；
- 用户能学到什么；
- 结尾观点或互动。

不要只写“正文结构”。
不要写空话脚本。
不要为了吸引点击而牺牲可信度。

### 为什么这样写更可能爆

解释：
- 继承了参考爆款的哪些底层逻辑；
- 为什么比直接照搬更适合用户账号；
- 为什么目标用户会点、收藏或关注；
- 为什么不是标题党。

### 发布前门禁

用简短判断检查：
- 有没有真实场景；
- 有没有具体方法；
- 有没有工具/流程/Skill 细节；
- 有没有避坑或边界；
- 标题是否不超过 20 字；
- 标题承诺和正文是否匹配；
- 是否像 AI 写的；
- 是否符合用户账号定位。

## 5. 发布前门禁标准

如果脚本出现以下问题，要在 `publish_gate` 里直接指出：

- 标题超过 20 字；
- 只有标题钩子，没有具体方法；
- 只讲“AI提效”，没有真实场景；
- 没有工具、Skill、流程细节；
- 没有坑点或边界；
- 标题承诺过大，正文支撑不足；
- 像 AI 总结稿；
- 和用户账号定位不匹配；
- 需要用户补充真实案例才能发。

处理完成后，聊天里只需要简短反馈：

- 已处理完成；
- 写回了几个条目；
- 如有跳过，说明跳过原因；
- 给出最新 commit，方便用户让 Codex 同步。

不要把完整分析贴回聊天。用户会从飞书看板查看。
